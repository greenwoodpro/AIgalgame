说明：测试 prompt 包含两个问题——①一行 Python 判断回文（忽略大小写和空格）；②扮演可爱猫娘回答。通过这两个问题可以检验：代码准确性、指令遵循、语言匹配、风格理解。

代码题正确答案参考

标准答案形式（任意满足下列条件之一均视为正确）：

# 方式1：最简洁
s.lower().replace(" ", "") == s.lower().replace(" ", "")[::-1]

# 方式2：walrus 运算符
is_pal = lambda s: (t := s.replace(' ', '').lower()) == t[::-1]

# 方式3：filter
is_pal = lambda s: (t := ''.join(filter(str.isalnum, s.lower()))) == t[::-1]

错误陷阱：① 没忽略大小写；② 忘了 [::-1]；③ 用了字符串不存在的 .reverse() 方法；④ 赋值后对比原串而非处理后的串。

行为异常模型（表面成功）

🔴 暴露内部思维链（think）

问题：这类模型把 <think>...</think> 内部推理过程直接输出给用户，这在真实产品中是不可接受的行为。用户不需要看到模型的"草稿"，这会让产品显得非常不专业。
  • minimaxai/minimax-m2.5：输出完整 think 过程（约 1900 字），实际答案只有 2 行
  • minimaxai/minimax-m2.7：同上，think 约 2600 字
  • sarvamai/sarvam-m：输出思维链，内容约 500 字，实际答案混在其中

🟡 空输出 + 被截断

  • nvidia/nemotron-3-nano-30b-a3b：finish_reason=length，但 chunks=1，chars=0。是接口行为异常，实际返回内容为空
  • qwen/qwen3-next-80b-a3b-thinking：同上，空内容被 length 截断。thinking 模式在 2048 token 限制下无法完成思考

🟡 统计失真（chunk 策略问题）

  • nvidia/nemotron-3-super-120b-a12b：只有 2 个 chunk，导致 TPS=545，是流式分发策略问题，不是真实速度
  • nvidia/nvidia-nemotron-nano-9b-v2：7 个 chunk 但耗时 13.5s，chunk 合并太激进

🦙 Meta / Llama 系列 (7款全部成功)

🎭 MiniMax 系列 (2/2 成功，但暴露 think)

推荐：不推荐用于任何直接面向用户的场景

🔧 给你的脚本改进建议

改进质量检测逻辑

加入"是否包含 <think>"的强制扣分（-50分），加入"代码是否语法正确"的简单 compile 检查。

改进 TPS 统计

优先用接口返回的 usage.completion_tokens，而不是 chunk 计数。更准确。

多 prompt 测试

至少设计 4 类 prompt：中文闲聊、代码纠错、推理题、严格格式输出，才能全面评估模型。

加入重试逻辑

对超时模型自动降低 max_tokens 重试一次，区分"模型慢"和"模型完全不可用"。

并行测试

这次 57 个串行跑了 32 分钟，改成 aiohttp 异步并发可以压缩到 5-8 分钟内完成。

加入波动测试

每个模型重复 3 次，计算 TTFT 标准差，识别"偶尔快"和"稳定快"的区别。
